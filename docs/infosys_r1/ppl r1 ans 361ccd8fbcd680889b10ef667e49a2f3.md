## Selection heuristics

The best heuristic model is this: Infosys is trying to separate candidates into three buckets — can’t solve, can solve standard DSE-level problems, and can solve harder SP-level problems consistently. ## Coding archetypes

The most repeated coding archetypes are: interval/state DP, tree path or subtree reasoning, graph traversal with constraints, binary-search-on-answer, sliding-window with invariants, and data-structure-augmented optimization like segment trees or heaps. The recurring failure mode is not “never seen this topic,” but “recognized the topic too late, coded too slowly, or failed on corner cases”.reddit+3

The best solving strategy is to classify each problem in the first 3–5 minutes into one of these buckets: direct standard pattern, standard pattern with twist, or custom reduction. If it is a standard pattern, execute immediately; if it is a twist, identify the invariant and build a minimal correct version before optimizing.linkedin+1

## Python survival sheet

Use Python for speed, but keep the code interview-safe and test-case friendly. Favor `sys.stdin.readline`, `collections.Counter`, `defaultdict`, `deque`, `heapq`, and `bisect` for fast implementation; avoid over-engineering with too many helper abstractions during the OA.geeksforgeeks+1

## Fastest templates

- Input: `import sys; input = sys.stdin.readline`.
- Frequency counting: `Counter` or `defaultdict(int)`.
- BFS: `deque`.
- Top-k / min-max selection: `heapq`.
- Ordered search: `bisect_left`, `bisect_right`.

## Pitfalls

- Off-by-one errors in 0/1 indexing.
- Forgetting hidden edge cases like empty array, one element, or duplicate values.
- Writing a recursive solution without checking recursion depth.
- Using brute force when the invariant suggests linear or nlog⁡nn \log nnlogn.reddit+1

## Aptitude warfare guide

For the general Infosys aptitude layer, the highest-yield topics are percentages, ratio, time-work, time-speed-distance, averages, probability, permutations-combinations, syllogisms, coding-decoding, data interpretation, and verbal grammar patterns. The practical strategy is to maximize elimination speed: solve easy arithmetic first, skip long DI until the end, and use option back-solving whenever equations become heavy.scribd+3

## Interview prediction system

The technical interview is usually a verification round: they may re-ask the coding logic, request complexity, ask you to improve the solution, and then move into OOP/DBMS/OS/CN/project depth. For SP, the pressure is higher: expect more probing on optimization choices, alternative data structures, and whether you can justify design tradeoffs. For DSE, the interview tends to be slightly more forgiving on depth but still checks whether you can code and explain your own work cleanly.glassdoor+3

Behavioral traps include claiming responsibility for code you cannot explain, vague project descriptions, and over-claiming advanced topics without implementation evidence. AI-cheating detection is mostly practical: interviewers can tell when a candidate cannot reproduce the logic, cannot change a solution under pressure, or cannot explain hidden-case behavior.linkedin+2

## Failure analysis

Most failures come from four causes: slow pattern recognition, incomplete edge-case handling, panic-driven syntax errors, and over-optimization that wastes time before a correct baseline exists. A frequent collapse point is solving one question beautifully and then losing the rest of the paper; the better strategy is to secure 1 full and 1 partial early, then push for the third.reddit+3

The anti-failure protocol is simple: first 10 minutes classify all 3 problems, first pass write the easiest complete solution, second pass upgrade the partial one, final pass clean corner cases and optimize. This is the highest-probability way to convert time into selection signal under this exam style.reddit+2
