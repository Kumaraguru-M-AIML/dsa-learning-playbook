### CATEGORY: MUST MASTER (25 Questions)

### Dynamic Programming (SP Differentiator)

1. **0/1 Knapsack** — *Recognition:* "subset with maximum value under weight constraint" — *Pattern:* 2D DP, dp[i][w] = max(dp[i-1][w], dp[i-1][w-wt[i]] + val[i]) — *Edge:* 0-index vs 1-index, large W
2. **Longest Common Subsequence** — *Recognition:* "longest subsequence common to two strings" — *Pattern:* 2D DP, dp[i][j] — *Edge:* empty strings
3. **Longest Increasing Subsequence** — *Recognition:* "longest strictly increasing subsequence" — *Pattern:* O(n²) DP or O(n log n) patience sorting — *Edge:* duplicate elements
4. **Coin Change (Min Coins)** — *Recognition:* "minimum coins to make amount" — *Pattern:* 1D DP, dp[i] = min(dp[i], dp[i-coin]+1) — *Edge:* impossible amounts
5. **Coin Change 2 (Ways)** — *Recognition:* "number of ways to make amount" — *Pattern:* 1D DP with coin outer loop — *Edge:* overflow, use modulo
6. **Edit Distance** — *Recognition:* "minimum operations to convert string1 to string2" — *Pattern:* 2D DP with insert/delete/replace — *Edge:* empty strings
7. **Subset Sum Problem** — *Recognition:* "can we form sum K from array subset?" — *Pattern:* Boolean 1D DP — *Edge:* sum = 0, large K
8. **Partition Equal Subset Sum** — *Recognition:* "can array be partitioned into two equal-sum subsets?" — *Pattern:* Subset sum with target = total/2
9. **Minimum Path Sum (Grid)** — *Recognition:* "minimum cost path from top-left to bottom-right" — *Pattern:* 2D DP, dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])
10. **Unique Paths / Unique Paths II** — *Recognition:* "number of ways to reach bottom-right" — *Pattern:* Combinatorics or DP — *Edge:* obstacles

### 6.2 Python Optimization for Infosys OA

1. **Always use `sys.stdin.readline`** — Standard `input()` is ~10x slower for large inputs.
2. **Prefer `list` over `deque` for random access** — deque for queue operations only.
3. **`@lru_cache` is your secret weapon for DP** — One line for memoization, zero bugs.
4. **Use `collections.Counter` for frequency problems** — O(n) one-liner.
5. **Python's `bisect` module** — Binary search without implementing from scratch.
6. **String concatenation: Use `''.join()`** — Never `+` in a loop (O(n²) vs O(n)).
7. **Set lookups are O(1)** — Prefer over list `.count()` or `in` on large lists.
8. **Avoid recursion for simple DFS on large graphs** — Iterative stack/deque is safer.

### 8.1 High-Probability Topics & Shortcuts

### Percentages

- **Shortcut:** 25% increase → multiply by 1.25; 20% decrease → multiply by 0.80
- **Successive changes:** a% up then b% down → net = a + b + (a×b)/100 (sign: up = +, down = -)

### Time-Speed-Distance

- **Shortcut:** Average speed for equal distances = 2ab/(a+b)
- **Relative speed:** Same direction = difference; Opposite = sum

### Permutation & Combination

- **Arrangement (P):** Order matters → nPr = n!/(n-r)!
- **Selection (C):** Order irrelevant → nCr = n!/(r!×(n-r)!)
- **Trigger word:** "committee", "team", "group" → Combination

### Probability

- **Simple:** P(Event) = Favorable Outcomes / Total Outcomes
- **At least one:** 1 - P(none occurring)
- **Independent events:** P(A ∩ B) = P(A) × P(B)

### Number System

- **Divisibility by 3:** Sum of digits divisible by 3
- **Divisibility by 4:** Last two digits divisible by 4
- **Divisibility by 8:** Last three digits divisible by 8
- **Divisibility by 9:** Sum of digits divisible by 9

## 10. RED TEAM FAILURE ANALYSIS — "ANTI-FAILURE PROTOCOL"

### 10.1 Psychological Collapse Points

| Failure Mode | Root Cause | Prevention |
| --- | --- | --- |
| **Blank Mind Effect** | Anxiety + lack of pattern recognition | Pre-memorize 5 template starters (see Python sheet) |
| **Coding Panic** | Seeing a Hard problem first | Always scan ALL problems first. Solve easiest → hardest |
| **Time Sink** | Perfectionism on one problem | 60-min hard limit per problem; move on and return |
| **Test Case Obsession** | Debugging edge cases endlessly | Get core logic right first; edge cases last 10 min |
| **Syntax Paralysis** | Forgetting Python constructs | Memorize survival sheet; use Python's forgiving syntax |

### 10.2 Over-Optimization Trap

**Symptom:** Implementing O(n) solution with complex data structures when O(n²) passes 70% of test cases.
**Rule:** **First make it work (brute force), then make it fast (optimize).** Partial credit beats zero credit.

### 10.3 Plagiarism Detection Risks

- **Wingspan proctoring** detects screen switching, background apps, and suspicious patterns
- **Code similarity checks** — Even variable renaming is detected
- **Consequence:** Permanent blacklisting, not just rejection
- **Rule:** Never attempt. Your own partial solution > perfect copied solution.

### 10.4 AI Dependence Collapse

**Risk:** Candidates who rely on ChatGPT/LLMs during preparation fail to develop independent problem-solving muscles.
**Protocol:** Practice with no assistance for 2-3 hour stretches. Simulate real OA conditions.

### During-Test Battle Plan

```
T+0 min   : Read ALL problems first (5 min max). Classify: Easy, Medium, Hard.
T+5 min   : Start with the EASIEST problem.
T+25 min  : Easy problem solved with all test cases passed. Move on.
T+30 min  : Start Medium problem.
T+75 min  : Medium problem solved (or at 80%+ test cases). Move on.
T+80 min  : Start Hard problem. Aim for brute force first.
T+130 min : Hard problem partial solution (50%+ test cases).
T+130-170 min : Return to any incomplete problems. Optimize. Debug edge cases.
T+170-180 min : Final scan. Check all test cases. Submit.
```

### Golden Rules During Test

1. **Never spend >60 minutes on one problem** — Mental reset is critical
2. **Brute force > No solution** — Always submit something that compiles
3. **Test cases tell you what's wrong** — Read failure outputs carefully
4. **If stuck for 15 min on a bug** — Take a deep breath, comment out, rewrite the logic block fresh
5. **Python tip:** Use `print()` debug statements strategically, then remove before final submit
