**Q1. Longest Common Subsequence (LCS)**

- **Pattern:** 2D DP, **`dp[i][j] = LCS of s1[:i], s2[:j]`**
- **Recognition trigger:** Two strings, find longest shared subsequence
- **Optimal Python:**

```
Python
def lcs(s1, s2):
    m, n= len(s1),len(s2)
    dp= [[0]*(n+1)for _in range(m+1)]
    for iin range(1, m+1):
        for jin range(1, n+1):
            dp[i][j]= dp[i-1][j-1]+1 if s1[i-1]==s2[j-1]else max(dp[i-1][j], dp[i][j-1])
    return dp[m][n]
```

- **Time complexity:** O(m×n)
- **Common mistake:** Forgetting to initialize base case (row 0 and column 0)
- **Edge case:** Empty string inputs → return 0

---

**Q2. 0/1 Knapsack**

- **Pattern:** 2D DP, items have weight/value, maximize value ≤ capacity W
- **Recognition trigger:** N items, W limit, maximize/minimize value
- **Optimal Python:**

```
Python
def knapsack(wt, val, W, n):
    dp= [[0]*(W+1)for _in range(n+1)]
    for iin range(1, n+1):
        for win range(W+1):
            dp[i][w]= dp[i-1][w]
            if wt[i-1]<= w:
                dp[i][w]= max(dp[i][w], dp[i-1][w-wt[i-1]]+ val[i-1])
    return dp[n][W]
```

- **Time complexity:** O(n×W)
- **Common mistake:** 1D vs 2D DP confusion; iterating w forward in 1D version causes double-counting
- **Edge case:** W=0 → return 0; single item heavier than W

**Q3. Longest Increasing Subsequence (LIS)**

- **Pattern:** DP with binary search optimization
- **Recognition trigger:** Array, find longest sequence where each next element > previous
- **Optimal Python (O(n log n)):**

```
Python
import bisect
def lis(nums):
    tails= []
    for nin nums:
        pos= bisect.bisect_left(tails, n)
        if pos== len(tails): tails.append(n)
        else: tails[pos]= n
    return len(tails)
```

- **Time complexity:** O(n log n)
- **Common mistake:** Using O(n²) DP when n is large causes TLE
- **Edge case:** All equal elements → LIS = 1

**Q5. BFS Shortest Path in Unweighted Graph**

- **Pattern:** Standard BFS with distance tracking
- **Recognition trigger:** Unweighted graph, minimum steps/hops between nodes
- **Optimal Python:**

```
Python
from collectionsimport deque
def bfs_shortest(graph, start, end, n):
    dist= [-1]* n
    dist[start]= 0
    q= deque([start])
    while q:
        node= q.popleft()
        for nbin graph[node]:
            if dist[nb]== -1:
                dist[nb]= dist[node]+ 1
                q.append(nb)
    return dist[end]
```

- **Time complexity:** O(V+E)
- **Common mistake:** Not marking visited before appending to queue → duplicate processing
- **Edge case:** Disconnected graph → return -1; start==end → return 0

**Q7. Sliding Window Maximum (Monotonic Deque)**

- **Pattern:** Deque-based sliding window
- **Recognition trigger:** Find max/min in every window of size k
- **Optimal Python:**

```

from collectionsimport deque
def maxSlidingWindow(nums, k):
    dq, res= deque(), []
    for i, nin enumerate(nums):
        while dqand nums[dq[-1]]< n: dq.pop()
        dq.append(i)
        if dq[0]== i- k: dq.popleft()
```

- **Recognition trigger:** Weighted graph, single source shortest path

```
def dijkstra(graph, src, n):

    dist[src]= 0
    heap= [(0, src)]

        if d> dist[u]:continue

            if dist[u]+w< dist[v]:
```

```
from collectionsimport deque

# BFS Template
def bfs(start, graph):
    visited= {start}
    queue= deque([start])
    while queue:
        node= queue.popleft()
        for neighborin graph[node]:
            if neighbornot in visited:
                visited.add(neighbor)
                queue.append(neighbor)

# DFS Iterative Template
def dfs(start, graph):
    visited= set()
    stack= [start]
    while stack:
        node= stack.pop()
        if nodein visited:continue
        visited.add(node)
        for neighborin graph[node]:
            if neighbornot in visited:
                stack.append(neighbor)

# Grid BFS
def grid_bfs(grid, sr, sc):
    R, C= len(grid),len(grid[0])
    visited= [[False]*Cfor _in range(R)]
    q= deque([(sr, sc,0)])
    visited[sr][sc]= True
    while q:
        r, c, d= q.popleft()
        for dr, dcin [(0,1),(0,-1),(1,0),(-1,0)]:
            nr, nc= r+dr, c+dc
            if 0<=nr<Rand 0<=nc<Cand not visited[nr][nc]and grid[nr][nc]!='#':
                visited[nr][nc]= True
```

**A. TIME, SPEED & DISTANCE (15% of aptitude)**

**Core formula:** Distance = Speed × Time

**Shortcut arsenal:**

- Two objects meeting: **`Time = Total distance / (Sum of speeds)`** if moving toward each other
- Two objects moving same direction: **`Time = Distance between / (Speed diff)`**
- Average speed (equal distances): **`2ab/(a+b)`** — NOT arithmetic mean
- Relative speed in circular track: Meeting time = Track length / (relative speed)

**Example trap:** [**5**](https://testbook.com/infosys)Anita goes to college at 20 km/h and reaches 4 minutes late. Next time she goes at 25 km/h and reaches 2 minutes earlier. What is the distance?

**Solution template:** Let distance = d. **`d/20 - d/25 = 6/60`** → **`d(25-20)/(20×25) = 1/10`** → **`d = 10 km`**

- Person A's work/day = LCM/time_A

# **PHASE 9 — RED TEAM FAILURE ANALYSIS**

# **═══════════════════════════════════════════════**

## **9.1 Why Candidates Fail — Root Cause Map**

### **🚨 Failure Mode 1: Q1 Time Overrun → Cascade Collapse**

**Pattern:** Candidate gets stuck on Q1 (supposedly easy). Spends 60+ minutes. No time for Q2/Q3. **Root cause:** Over-engineered solution attempt on simple problem. Looking for "trick" where none exists. **Anti-failure protocol:** If Q1 takes more than 30 minutes, accept partial score and MOVE ON. Partial Q2 is worth more than perfect Q1.

---

### **🚨 Failure Mode 2: Output Format Mismatch**

[**25**](https://www.educative.io/blog/infosys-coding-interview-questions)

Even logically correct code can fail if spacing, newlines, or extraneous print statements do not match the expected format exactly.

**Root cause:** Candidate solves the problem but fails all test cases due to extra spaces, wrong newline, or debugging prints left in. **Anti-failure protocol:** Always print exactly what the problem says. Remove all debug prints. Test sample input/output manually before submitting.

---

### **🚨 Failure Mode 3: TLE (Time Limit Exceeded) on Hard Problem**

**Pattern:** Candidate submits O(n²) solution for large n. Partial test cases pass; most fail. **Root cause:** Not thinking about time complexity before coding. **Anti-failure protocol:** BEFORE writing code, estimate: "Will O(n²) pass for n=10⁵? No. I need O(n log n) or O(n)."

- n ≤ 10⁴: O(n²) okay
- n ≤ 10⁵: O(n log n) required
- n ≤ 10⁶: O(n) required

---

### **🚨 Failure Mode 4: Recursion Stack Overflow**

**Pattern:** DFS/recursive DP on large input → Python's default recursion limit hits (1000) → Runtime error. **Root cause:** Not setting recursion limit or not using iterative versions. **Anti-failure protocol:**

```
Python
import sys
sys.setrecursionlimit(10**6)# Put at TOP of every solution file
```

Or convert recursion to iterative with explicit stack.

---

### **🚨 Failure Mode 5: Blank Mind Effect on Novel Problems**

**Pattern:** See a problem that doesn't match memorized patterns → panic → blank mind → submit nothing. **Root cause:** Memorized solutions rather than patterns. **Anti-failure protocol:** Develop pattern recognition, not solution memorization. When stuck:

1. Brute force first (write it)
2. Identify bottleneck (what operation is repeated?)
3. Apply one optimization (memoization? sorting? prefix?)
4. Submit partial brute force if needed — some test cases pass

---

### **🚨 Failure Mode 6: AI Dependency Collapse**

**Pattern:** Candidate relied on AI/ChatGPT during practice. During exam (no AI), cannot write basic code. **Root cause:** Outsourced thinking to AI. **Anti-failure protocol:** Every practice problem must be solved INDEPENDENTLY first. AI review only after your solution. The exam environment has no AI.

---

### **🚨 Failure Mode 7: Interview — Resume Attack Exposure**

**Pattern:** Candidate lists technologies they don't fully understand. Interviewer asks one deep question → cascade failure of confidence. **Root cause:** Padding resume with tech buzzwords.

[**15**](https://www.glassdoor.co.in/Interview/Infosys-Digital-Specialist-Engineer-Interview-Questions-EI_IE7927.0,7_KO8,35.htm)

The interviewers also discussed the candidate's projects and asked them to explain implementation details clearly.

**Anti-failure protocol:** Only list what you can explain for 15+ minutes under pressure. One deep project > five shallow ones.

---

### **🚨 Failure Mode 8: Psychological Collapse Under Observation (Live Coding)**

**Pattern:** Candidate knows the algorithm but cannot code while interviewer watches.

[**27**](https://leetcode.com/discuss/post/7302950/)

At least 1 coding question must be solved completely in front of the interviewer on the Wingspan portal.

**Root cause:** Never practiced explaining solutions aloud while coding. **Anti-failure protocol:** During final prep week, practice ALL problems by explaining your approach aloud as you code. This builds the "observed coding" neural pathway.

---

### **🚨 Failure Mode 9: Login Window Miss**

[**8**](https://prepinsta.com/infosys-sp-and-dse/recruitment-process/)

You must login within 15 minutes of the start of the test slot time as the login window will remain open only for 15 minutes once the test time begins.

**Root cause:** Candidate logs in late due to technical issues. **Anti-failure protocol:** Have system ready 30 minutes before test. Test browser compatibility 24 hours before. Have backup internet (hotspot). Know the exact login URL in advance.
